# Module 03: Database & Schema Design

---

## 1. Purpose

### Why This Module Exists
The **Database & Schema Design** module represents the persistent memory of TaskFlow. In system design and database interviews, engineers are judged on their ability to structure relational data, enforce integrity constraints, design optimal indexes, prevent anomalies, and design migrations that can run safely under high read/write loads.

### What Problem It Solves
TaskFlow manages complex nested dependencies.
* A user belongs to multiple workspaces with different roles (RBAC).
* A workspace has multiple projects, each project has a single board, which contains columns, which contain tasks, which have sub-checklist items, comments, file attachments, and logged hours.
* If a project is deleted, we must not leave orphaned tasks or attachments (data integrity).
* If two members join the same workspace, we must prevent duplicate rows (uniqueness constraints).

Enforcing these boundaries at the database level guarantees data consistency, prevents orphaned records, and ensures high query performance.

### How It Interacts With Other Modules
The database schema defines the type-safe classes generated by the **Prisma ORM**.
* The **RBAC Engine** queries the `workspace_members` join table to check roles.
* The **Time Log & Billing controller** runs complex query aggregations on the `time_logs` table.
* The **Gantt Calendar view** reads `createdAt` and `dueDate` from the `tasks` table.

```
[ User ] <--- (1:N) ---> [ WorkspaceMember ] <--- (N:1) ---> [ Workspace ]
                                                                  |
                                                                (1:N)
                                                                  v
                                                             [ Project ]
                                                                  |
                                                                (1:1)
                                                                  v
                                                              [ Board ]
                                                                  |
                                                                (1:N)
                                                                  v
                                                              [ Column ]
                                                                  |
                                                                (1:N)
                                                                  v
                                                               [ Task ] <--- (1:N) ---> [ TimeLog ]
```

### Real-World Analogy
Think of the database schema as a municipal city registry.
* The **User table** is the list of citizen identity profiles.
* The **WorkspaceMember table** is the business registry listing who works at which company and their specific role (Manager, Employee, Auditor).
* **Foreign keys** are physical street addresses linking properties. A house (Task) cannot exist without a street (Column), and a street cannot exist without a neighborhood (Board). If a neighborhood is completely demolished (deleted), all its streets and houses are cleared automatically.

## 3. Detailed Workflow

Let us trace how the database executes and validates a request to log hours on a task: **A user logs 2 hours on Task X.**

### Execution Sequence
1. **Constraint Check**:
   * The application issues an `INSERT` query into the `time_logs` table: `{ taskId: "task-uuid", userId: "user-uuid", durationSeconds: 7200, description: "Fixing security bugs" }`.
   * The database verifies that the `taskId` matches a valid row in the `tasks` table and the `userId` matches a valid row in the `users` table (Foreign Key Constraint Validation).
2. **Indexing**:
   * PostgreSQL inserts the record and updates the index on `time_logs(taskId)` and `time_logs(userId)`. This ensures that subsequent queries fetching logs for a task run in $O(\log N)$ instead of reading the entire table.
3. **Cascades**:
   * If a user subsequently deletes Task X, PostgreSQL references the foreign key definition on the `TimeLog` model: `task Task @relation(fields: [taskId], references: [id], onDelete: Cascade)`.
   * The database automatically deletes all rows in the `time_logs` table containing `taskId: "task-uuid"`.
4. **Error Handling**:
   * **Foreign Key Violation**: If the client submits a non-existent `taskId`, the insert fails, throwing a `23503` SQL state exception. Prisma translates this into a `P2003` client exception, prompting the backend to return `404 Not Found`.

## 4. Classes (Model Representation)

Prisma generates TypeScript interfaces matching our database schema. Below are key model representations:

### `WorkspaceMember` (Join Model)
* **Purpose**: Maps the many-to-many relationship between Users and Workspaces with specific role access.
* **Key Fields**:
  * `workspaceId`: `string` (Foreign key to Workspace)
  * `userId`: `string` (Foreign key to User)
  * `role`: `Role` (Enum: `OWNER`, `ADMIN`, `MEMBER`, `VIEWER`)
  * `joinedAt`: `DateTime`
* **Constraints**: Composite primary key `@@id([workspaceId, userId])`.
* **Why This Design**: Using a composite primary key ensures a user can only have a single membership row per workspace, preventing duplicate role records.

## 5. Functions

Since database structures are declarative, "Functions" are represented by complex query scripts executed by our backend controllers.

### `getProjectBillingData(projectId)`
* **Purpose**: Aggregates time logs and groups them by task and user for invoicing.
* **Parameters**:
  * `projectId`: `string`
* **Return Value**: An aggregated object containing total hours, user breakdowns, and task breakdowns.
* **Time Complexity**: $O(K \log L)$ where $K$ is the number of time logs and $L$ is the number of tasks/users.
* **Execution (Prisma Query)**:
  ```typescript
  const logs = await prisma.timeLog.findMany({
    where: {
      task: {
        column: {
          board: {
            projectId
          }
        }
      }
    },
    include: {
      task: { select: { id: true, title: true } },
      user: { select: { id: true, name: true, email: true, avatarUrl: true } }
    }
  });
  ```
* **Why Written This Way**: This query traverses the relation tree (`TimeLog -> Task -> Column -> Board -> Project`) in a single step using SQL joins, eliminating the need to execute separate queries for each relation layer.

## 6. Architecture Discussion

### Database Scalability & Normalization
* **3NF Compliance**: The schema is normalized to the Third Normal Form (3NF) to eliminate redundant data. User data is isolated to the `users` table; workspaces only reference `ownerId` instead of duplicating owner profiles.
* **Denormalization Trade-offs**: In high-scale production systems, we might denormalize fields (like storing a `taskCount` on the `Column` model) to avoid running `COUNT` queries on every board render. However, this introduces synchronization complexity.
* **Index Design**: We explicitly define indexes on fields commonly used in filters or sort queries (like `workspaceId` on `Project`, `boardId` on `Column`, and `taskId` on `TimeLog`).

## 7. Interview Questions

### Easy (15)

#### 1. What is a primary key?
* **Answer**: A unique identifier for a row in a database table. No two rows can share the same primary key, and it cannot be null.
* **Explanation**: In TaskFlow, most tables use UUID strings as their primary keys to guarantee uniqueness across distributed systems.
* **Follow-up**: *What is the advantage of UUIDs over auto-incrementing integers?* UUIDs can be generated by clients offline without risking collisions, whereas integers require the database to coordinate the next sequence number.
* **Common Mistakes**: Stating that a table can have multiple independent primary keys.

#### 2. What is a foreign key?
* **Answer**: A column or group of columns in one table that references the primary key of another table, enforcing referential integrity.
* **Explanation**: In the `tasks` table, `columnId` is a foreign key referencing `id` in the `columns` table.
* **Follow-up**: *What happens if we try to insert a task with a non-existent columnId?* The database rejects the insertion with a foreign key violation error.
* **Common Mistakes**: Thinking foreign keys automatically create database indexes in all SQL engines (PostgreSQL requires explicit indexing).

#### 3. What is a many-to-many relationship, and how is it resolved in a database?
* **Answer**: A relationship where multiple records in Table A associate with multiple records in Table B. It is resolved using a junction (join) table.
* **Explanation**: In TaskFlow, Users and Workspaces share a many-to-many relationship, resolved by the `WorkspaceMember` junction table.
* **Follow-up**: *Can a junction table store additional properties?* Yes, our `WorkspaceMember` table stores the user's role and join date.
* **Common Mistakes**: Creating array columns inside Table A to reference Table B keys instead of using a proper junction table.

#### 4. What is a unique constraint?
* **Answer**: A database rule that prevents duplicate values from being written to specific columns.
* **Explanation**: The `email` column in the `users` table has a unique constraint, preventing two users from registering with the same email.
* **Follow-up**: *What happens if a unique constraint check fails?* The database returns a unique key constraint violation exception.
* **Common Mistakes**: Checking for uniqueness only via application code queries, which fails under concurrent registrations.

#### 5. What are PostgreSQL Enums?
* **Answer**: Custom data types that restrict values to a predefined list.
* **Explanation**: In our schema, `Role` is an enum with values: `OWNER`, `ADMIN`, `MEMBER`, `VIEWER`.
* **Follow-up**: *What is the benefit of enums over string columns?* Enums consume less storage space and enforce validation checks at the database level.
* **Common Mistakes**: Believing enums are easy to modify in production without executing schema migrations.

#### 6. What does `onDelete: Cascade` mean?
* **Answer**: A constraint behavior that automatically deletes dependent child records when the parent record is deleted.
* **Explanation**: If a project is deleted, all its tasks are deleted automatically.
* **Follow-up**: *When should we avoid cascade deletes?* For critical logs or financial records where audit trails must be preserved. We use `onDelete: SetNull` or block deletions instead.
* **Common Mistakes**: Misunderstanding the flow of cascade deletes and assuming deleting a child deletes the parent.

#### 7. What is a database index?
* **Answer**: A data structure (usually a B-Tree) that improves data retrieval speeds at the cost of additional write overhead and storage space.
* **Explanation**: Creating an index on `task(columnId)` allows the database to find tasks belonging to a column without scanning the entire table.
* **Follow-up**: *Why does it slow down writes?* Because the database must rebuild or update the index structure on every insert, update, and delete.
* **Common Mistakes**: Indexing every single column in a table without analyzing query patterns.

#### 8. What is the difference between SQL and NoSQL databases?
* **Answer**: SQL databases are relational, use structured schemas, and guarantee ACID compliance. NoSQL databases are non-relational, schema-flexible, and optimized for horizontal scaling.
* **Explanation**: We choose SQL (PostgreSQL) for TaskFlow because our data is highly relational and requires transaction safety.
* **Follow-up**: *Could we use MongoDB for TaskFlow?* Yes, but managing workspace member permissions and task relations would require writing complex join emulation logic in application code.
* **Common Mistakes**: Stating that SQL databases cannot scale or that NoSQL databases do not support relationships.

#### 9. What is a database migration?
* **Answer**: A version-controlled set of SQL scripts used to update database schemas consistently across environments.
* **Explanation**: When we add a model (like `TimeLog`), Prisma generates a migration script to update the database schema without losing existing data.
* **Follow-up**: *Where are migrations stored in the codebase?* Inside the `prisma/migrations` folder.
* **Common Mistakes**: Manually editing the database schema in production instead of using migrations.

#### 10. What is a composite primary key?
* **Answer**: A primary key composed of two or more columns that together guarantee row uniqueness.
* **Explanation**: `WorkspaceMember` uses `@@id([workspaceId, userId])` as its composite primary key.
* **Follow-up**: *Can a user have multiple roles in the same workspace?* No, the composite key restricts them to a single row in the junction table.
* **Common Mistakes**: Creating a separate `id` column on junction tables when a composite key is more appropriate.

#### 11. What is referential integrity?
* **Answer**: A database state where all foreign key references are valid, ensuring no columns point to non-existent records.
* **Explanation**: The database maintains referential integrity by blocking deletions of parents that have active dependencies if cascade is not configured.
* **Follow-up**: *How is it enforced?* Using foreign key constraints.
* **Common Mistakes**: Assuming application code is sufficient to guarantee referential integrity.

#### 12. What does ACID stand for?
* **Answer**: Atomicity, Consistency, Isolation, Durability. The core properties that guarantee database transactions are processed reliably.
* **Explanation**: In TaskFlow, updating card indices must satisfy ACID properties to prevent corrupt board layouts.
* **Follow-up**: *What is Atomicity?* The "all or nothing" execution rule: if one query in a transaction fails, the entire transaction rolls back.
* **Common Mistakes**: Thinking ACID properties are only required in financial banking systems.

#### 13. What is the difference between a table scan and an index scan?
* **Answer**: A table scan reads every page of a database table to find matching rows. An index scan queries the index structure to locate rows directly.
* **Explanation**: A table scan is $O(N)$ (slow), while an index scan is $O(\log N)$ (fast).
* **Follow-up**: *When will the database perform a table scan even if an index exists?* If the table is very small, as reading the table directly can be faster than loading index pages.
* **Common Mistakes**: Assuming the database optimizer always uses indexes.

#### 14. What is a NULL value in SQL?
* **Answer**: A marker indicating that a data value does not exist in the database.
* **Explanation**: The `dueDate` field on `Task` is nullable because tasks don't always require due dates.
* **Follow-up**: *How do you query for null values?* Using `IS NULL` or `IS NOT NULL`.
* **Common Mistakes**: Treating NULL as an empty string `""` or zero `0` (NULL represents the absence of value).

#### 15. What is a database transaction?
* **Answer**: A sequence of database operations executed as a single logical unit of work.
* **Explanation**: If one query in the transaction fails, all changes are rolled back.
* **Follow-up**: *How do we execute transactions in Prisma?* Using `prisma.$transaction([query1, query2])`.
* **Common Mistakes**: Performing external HTTP API calls inside a database transaction block.

### Medium (20)

#### 1. How would you design a database schema to support task comments and user mentions?
* **Interviewer's Intent**: To verify relational modeling skills and clean schema design for collaborative features.
* **Answer**: We create a `Comment` model referencing `taskId` and `userId` via foreign keys. To support user mentions, we create a many-to-many junction table `CommentMention` linking comments to mentioned users.
* **Why Interviewer Asks**: Real-time collaborative features require clean database structures to prevent performance degradation as comment volumes grow.
* **Common Mistakes**: Parsing comment text for mentions on every query instead of storing structured mention relationships in a table.
* **Follow-up**: *How do you index the comment table?* We index `comment(taskId, createdAt)` to fetch task comments sorted by time.
* **Production Example**: standard schema design used by Jira or GitHub issues.

#### 2. What is database normalization, and why do we normalize up to 3NF?
* **Interviewer's Intent**: To check database theory knowledge and data integrity design skills.
* **Answer**: Normalization is the process of structuring a relational database to reduce data redundancy and improve data integrity. Third Normal Form (3NF) requires that all columns depend only on the primary key, eliminating transitive dependencies.
* **Why Interviewer Asks**: Under-normalized databases suffer from write anomalies, where updating a value (like user name) requires modifying multiple tables.
* **Common Mistakes**: Over-normalizing high-throughput tables where denormalization would improve read speeds.
* **Follow-up**: *What is an example of transitive dependency?* If Task stored `columnId` and `boardId`, since `columnId` determines `boardId`, storing both on Task violates 3NF.
* **Production Example**: Splitting workspaces, members, and users into distinct tables.

#### 3. How do you handle schema changes on large tables without locking the database?
* **Interviewer's Intent**: To check production engineering capabilities and zero-downtime database administration skills.
* **Answer**: Altering columns or adding non-nullable fields on tables with millions of rows locks the table, causing API timeouts. We handle this by adding new columns as nullable, deploying code that writes to both columns, backfilling old rows in batches, and finally dropping the old column.
* **Why Interviewer Asks**: Applying migrations directly on large production tables can cause hours of downtime if table locks block API writes.
* **Common Mistakes**: Running `ALTER TABLE ADD COLUMN NOT NULL DEFAULT` directly on live tables.
* **Follow-up**: *How do you create an index on a large table without blocking writes in PostgreSQL?* Using the `CONCURRENTLY` keyword (`CREATE INDEX CONCURRENTLY`).
* **Production Example**: Executing migrations in sequential steps: expand first, deploy code, check stability, contract schema.

#### 4. What is the difference between INNER JOIN, LEFT JOIN, and outer joins?
* **Interviewer's Intent**: To verify SQL query building blocks.
* **Answer**: 
  * `INNER JOIN` returns rows only when there is a match in both tables.
  * `LEFT JOIN` returns all rows from the left table and matched rows from the right table (filling with NULL on misses).
  * `FULL OUTER JOIN` returns all rows when there is a match in either table.
* **Why Interviewer Asks**: Writing incorrect joins returns wrong data or impacts database retrieval performance.
* **Common Mistakes**: Using INNER JOIN when fetching optional relationships (like Task due dates or assignees), which filters out valid parent records.
* **Follow-up**: *Which join does Prisma use when executing `include`?* Typically a LEFT JOIN or batch SELECT queries depending on the query planner.
* **Production Example**: Fetching tasks and optionally including their file attachments using a LEFT JOIN.

#### 5. How do composite indexes work, and why does column order matter?
* **Interviewer's Intent**: To check database indexing proficiency and query optimization skills.
* **Answer**: A composite index is an index on multiple columns (e.g. `idx_task_col_priority` on `Task(columnId, priority)`). Column order matters because the index is sorted by the first column first, then the second. A query filtering only by the second column cannot utilize the index.
* **Why Interviewer Asks**: Creating composite indexes in the wrong order wastes storage and fails to optimize target queries.
* **Common Mistakes**: Assuming an index on `(A, B)` automatically optimizes queries filtering only by `B`.
* **Follow-up**: *What is the left-prefix rule?* The rule stating that a composite index can only be used if the query filters by the leftmost prefix of index columns.
* **Production Example**: Creating a composite index on `WorkspaceMember(workspaceId, userId)` to optimize member lookup queries.

#### 6. What is the difference between Prisma's `implicit` and `explicit` many-to-many relationships?
* **Interviewer's Intent**: To evaluate ORM configuration decisions.
* **Answer**: Implicit many-to-many relationships let Prisma manage the junction table automatically. Explicit many-to-many relationships require us to define the junction model in our schema, allowing us to add custom fields to the relationship (like `role` or `joinedAt`).
* **Why Interviewer Asks**: Implicit relationships are simple but limit flexibility if requirements change later (e.g., needing to track when an assignee was added to a task).
* **Common Mistakes**: Using implicit relationships for core domain entities that will eventually require metadata tracking.
* **Follow-up**: *Which one did we use for TaskAssignees?* Implicit, as we only need to link users to tasks. We used explicit for `WorkspaceMember`.
* **Production Example**: Defining `WorkspaceMember` as an explicit model to support custom roles.

#### 7. How does PostgreSQL handle MVCC (Multi-Version Concurrency Control)?
* **Interviewer's Intent**: To check understanding of database storage engines and isolation mechanics.
* **Answer**: MVCC allows concurrent reads and writes without blocking. When a row is updated, PostgreSQL doesn't overwrite it; it writes a new version of the row. Readers continue reading the old version until the writing transaction commits.
* **Why Interviewer Asks**: MVCC is central to PostgreSQL performance. Understanding it helps diagnose database size bloat and vacuuming requirements.
* **Common Mistakes**: Assuming updates overwrite disk sectors directly.
* **Follow-up**: *What is the purpose of the VACUUM command?* To clean up dead row versions (garbage collection) created by MVCC updates.
* **Production Example**: Running autovacuum processes on database servers.

#### 8. What is database connection starvation, and how do you prevent it?
* **Answer**: Connection starvation happens when the application runs out of database connection handles, causing queries to queue and time out. We prevent it by configuring connection pooling, optimization of query speeds, and routing long-running queries to read replicas.
* **Why Interviewer Asks**: High traffic spikes can easily crash applications if the connection pool is not configured correctly.
* **Common Mistakes**: Increasing pool sizes indefinitely, which exhausts database server memory.
* **Follow-up**: *What tool acts as an external pooler for PostgreSQL?* PgBouncer.
* **Production Example**: Configuring `PgBouncer` in front of PostgreSQL database clusters.

#### 9. How do you design a schema to store audit logs efficiently?
* **Answer**: Audit logs are write-heavy and read-rare. We structure the table with minimal foreign key constraints, index only essential query fields (like `userId` or `action`), and use table partitioning (e.g. by month) to make archiving old logs simple.
* **Why Interviewer Asks**: Evaluates log aggregation and database partition design skills.
* **Common Mistakes**: Adding cascading deletes on audit logs, which wipes out history when parent objects are deleted.
* **Follow-up**: *How do we link logs to deleted entities?* By storing entity IDs as simple string fields instead of relational foreign keys.
* **Production Example**: Storing audit logs in partitioned PostgreSQL tables or exporting them to log streams.

#### 10. What is database sharding, and when should you implement it?
* **Answer**: Sharding partitions data horizontally across independent database servers (e.g. users A-M on Server 1, N-Z on Server 2). It should only be implemented when write volumes exceed the capacity of a single database server and replica scaling.
* **Why Interviewer Asks**: Sharding adds massive complexity. Senior engineers must understand the trade-offs before recommending it.
* **Common Mistakes**: Recommending sharding for systems that can be scaled simply by optimization of queries or vertical sizing.
* **Follow-up**: *What is the biggest drawback of sharding?* Running cross-shard queries and joins is slow and complex.
* **Production Example**: Multi-tenant architectures sharding data by workspace ID.

#### 11. What is the difference between a clustered and non-clustered index?
* **Answer**: A clustered index dictates the physical order of data rows on disk. A non-clustered index is a separate structure containing pointers to data rows.
* **Why Interviewer Asks**: Explains database storage engine organization.
* **Common Mistakes**: Assuming a table can have multiple clustered indexes (physical data can only be sorted in one order).
* **Follow-up**: *How does PostgreSQL handle this?* PostgreSQL uses heap-organized tables; it does not have native clustered indexes, but provides the `CLUSTER` command to sort tables physically on an index.
* **Production Example**: Optimizing primary key lookups.

#### 12. What are database transactions isolation levels?
* **Answer**: Isolation levels control the visibility of changes made by concurrent transactions. The standard levels are: Read Uncommitted, Read Committed, Repeatable Read, and Serializable.
* **Why Interviewer Asks**: Explains data safety trade-offs in concurrent environments.
* **Common Mistakes**: Defaulting to Serializable isolation for all queries, which degrades write throughput.
* **Follow-up**: *What is the default isolation level in PostgreSQL?* Read Committed.
* **Production Example**: Setting Serializable isolation on financial balance updates.

#### 13. How do you design a database schema to support project templates?
* **Answer**: We create template tables (e.g. `TemplateColumn`, `TemplateTask`) decoupled from live boards. When a project is created from a template, we copy the template rows into the live `columns` and `tasks` tables.
* **Why Interviewer Asks**: Evaluates configuration and copy design patterns.
* **Common Mistakes**: Linking live tasks directly to templates, which causes integrity errors if templates are modified.
* **Follow-up**: *How do we run this copy operation efficiently?* Using a single transaction with bulk insert statements.
* **Production Example**: Template generation engines in Jira or Trello.

#### 14. What are database savepoints, and when are they used?
* **Answer**: Savepoints are markers within a transaction that allow rolling back a subset of queries without aborting the entire transaction.
* **Why Interviewer Asks**: Explains advanced transaction rollback strategies.
* **Common Mistakes**: Thinking savepoints can be committed independently.
* **Follow-up**: *Does Prisma support savepoints?* Yes, Prisma uses savepoints under the hood for nested transactions.
* **Production Example**: Executing optional database updates within a larger transaction.

#### 15. How do you detect and resolve database deadlocks?
* **Answer**: Deadlocks happen when Transaction 1 holds Lock A and requests Lock B, while Transaction 2 holds Lock B and requests Lock A. PostgreSQL detects deadlocks automatically, aborts one of the transactions, and rolls it back. We prevent deadlocks by ensuring all queries update rows in the exact same index order.
* **Why Interviewer Asks**: Concurrency bugs like deadlocks degrade database availability under load.
* **Common Mistakes**: Retrying transactions without exponential backoff, which worsens deadlock loops.
* **Follow-up**: *How do you log deadlocks?* By configuring PostgreSQL's `log_lock_waits` and `deadlock_timeout` parameters.
* **Production Example**: Standardizing the order of row updates in transactions.

#### 16. What is table partitioning in PostgreSQL?
* **Answer**: Splitting a large table physically into smaller, manageable tables (partitions) while retaining a single query interface.
* **Why Interviewer Asks**: Essential for managing massive tables like logs or metrics.
* **Common Mistakes**: Partitioning small tables, which increases query overhead.
* **Follow-up**: *What are the common partitioning strategies?* Range partitioning (e.g. by date) and List partitioning (e.g. by category).
* **Production Example**: Partitioning log tables by month.

#### 17. How do you prevent SQL injection when using dynamic search filters in Prisma?
* **Answer**: Prisma handles parameter binding automatically. Even when using raw SQL, we use template literals that Prisma parses into parameterized inputs, ensuring values are bound to placeholders instead of concatenated directly.
* **Why Interviewer Asks**: SQL injection is a severe security risk.
* **Common Mistakes**: Concatenating input strings directly inside SQL query strings.
* **Follow-up**: *What method does Prisma provide for raw queries?* `prisma.$queryRaw`.
* **Production Example**: Building secure dynamic filter queries in search controllers.

#### 18. What is the difference between database replication lag and replication conflict?
* **Answer**: Replication lag is the delay between a write on the primary database and its sync to a read replica. A replication conflict happens when a write on the primary conflicts with a read query running on a replica.
* **Why Interviewer Asks**: Explains read-scaling synchronization challenges.
* **Common Mistakes**: Assuming read replicas are always in perfect real-time sync with the primary.
* **Follow-up**: *How do you mitigate replica lag issues?* By routing critical reads to the primary database.
* **Production Example**: Standard read-replica failover management.

#### 19. How do you design a schema to support soft deletes?
* **Answer**: We add a nullable `deletedAt` DateTime column to tables. Instead of deleting rows physically, we update `deletedAt` to the current timestamp. We filter out rows where `deletedAt` is not null in our application queries.
* **Why Interviewer Asks**: Soft deletes are common in enterprise applications to prevent accidental data loss.
* **Common Mistakes**: Forgetting to update unique constraints to account for soft-deleted rows.
* **Follow-up**: *How do you handle unique indexes with soft deletes?* By creating partial unique indexes that only target rows where `deletedAt` is null.
* **Production Example**: User deletion flows.

#### 20. What is database vacuuming, and why does PostgreSQL require it?
* **Answer**: PostgreSQL uses MVCC, meaning updates and deletes leave "dead" versions of rows on disk. Vacuuming reclaims this space and updates visibility maps, preventing disk space bloat and keeping queries fast.
* **Why Interviewer Asks**: Poor vacuum configuration causes database disk space exhaustion in write-heavy systems.
* **Common Mistakes**: Disabling autovacuum in production settings.
* **Follow-up**: *What is the difference between VACUUM and VACUUM FULL?* Standard VACUUM reclaims space for reuse within the table; VACUUM FULL physically rewrites the table on disk, reclaiming space to the OS but locking the table.
* **Production Example**: Monitoring autovacuum logs on PostgreSQL servers.

### Hard (20)

#### 1. How would you design a multi-tenant database strategy for a SaaS version of TaskFlow? Discuss trade-offs between Schema-per-tenant, DB-per-tenant, and Shared-DB.
* **Detailed Answer**:
  1. **DB-per-tenant**: Each client gets a dedicated database server.
     * *Pros*: Maximum isolation, compliance, and custom backup schedules.
     * *Cons*: High costs, resource waste for small tenants, and complex migrations.
  2. **Schema-per-tenant**: One database containing isolated schemas for each tenant.
     * *Pros*: Good isolation, migrations can be run per tenant.
     * *Cons*: PostgreSQL performance degrades when managing thousands of schemas.
  3. **Shared-Database (Row-Level Isolation)**: All tenants share tables, isolated by a `tenantId` (or `workspaceId`) column.
     * *Pros*: Low cost, simple query patterns, and easy resource scaling.
     * *Cons*: Risk of data leaks if query filters are missed, and "noisy neighbor" resource contention.
* **Deep Explanation**: For TaskFlow, we use the **Shared-Database** approach. To mitigate security risks, we enforce workspace filters in all queries and index `workspaceId` across all models.
* **Alternative Approach**: Hybrid model, where Enterprise tenants get dedicated databases, while Free/Pro tenants share a single database.
* **Production Example**: Slack uses a hybrid sharded database architecture, while Jira Cloud uses schema/database separation per instance.
* **Cross Questions**: *How do you prevent developer errors from leaking data across tenants in a shared database?* By implementing PostgreSQL **Row Level Security (RLS)** policies that enforce filters at the database engine level based on the active session user.

#### 2. How do you design and implement table partitioning for the `ActivityLog` table to handle 100M+ rows while maintaining $O(\log N)$ query times?
* **Detailed Answer**: We partition the `ActivityLog` table using **Range Partitioning** based on the `createdAt` timestamp, splitting tables by month (e.g. `activity_logs_2026_07`).
* **Deep Explanation**: 
  1. We create a master table without physical storage: `CREATE TABLE activity_logs (...) PARTITION BY RANGE (createdAt);`.
  2. We configure partition tables for specific ranges: `CREATE TABLE activity_logs_2026_07 PARTITION OF activity_logs FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');`.
  3. Queries filtering by `createdAt` utilize **partition pruning**, meaning the query planner only searches the index of the specific month's partition table, bypassing other indexes.
* **Alternative Approach**: Keeping a single monolithic table with index adjustments.
  * *Pros*: Simple, zero partition management code needed.
  * *Cons*: Index sizes exceed RAM capacity, causing disk-thrashing and query timeouts.
* **Production Example**: Log aggregation databases.
* **Cross Questions**: *What happens if an insert falls outside defined partition ranges?* The query fails. We must pre-create partitions using cron scripts or implement a default partition table to capture overflows.

#### 3. How do you prevent race conditions and write skew in a room-booking database schema? Relate this to TaskFlow assignees.
* **Detailed Answer**: Write skew happens when two transactions read the same data, make independent decisions, and write updates that violate business rules (e.g., two users attempt to assign the last slot on a task concurrently). We prevent this by setting the transaction isolation level to **Serializable** or using **Pessimistic Locking** (`SELECT ... FOR UPDATE`).
* **Deep Explanation**:
  ```sql
  -- Session 1
  SELECT count(*) FROM TaskAssignees WHERE taskId = 'X'; -- Returns 0
  -- Session 2 (Concurrent)
  SELECT count(*) FROM TaskAssignees WHERE taskId = 'X'; -- Returns 0
  -- Both sessions proceed to insert rows, violating a constraint limit of 1 assignee.
  ```
  Using `SELECT ... FOR UPDATE` locks the task row, forcing Session 2 to wait until Session 1 completes its transaction.
* **Alternative Approach**: Unique constraint indexing.
  * *Pros*: High performance, zero lock waits.
  * *Cons*: Limited to static uniqueness constraints; cannot handle dynamic logical rules.
* **Production Example**: Ticketing systems or hotel room booking engines.
* **Cross Questions**: *Does Serializable isolation throw errors on write conflicts?* Yes, PostgreSQL aborts one of the transactions, requiring the application to retry.

#### 4. How would you design a database schema to support real-time collaborative spreadsheets?
* **Detailed Answer**: We must model cells as atomic units. Instead of storing columns as table attributes, we store them as coordinates in a `Cell` table: `Cell(sheetId, row, col, value, version)`. Concurrent edits are processed using OT (Operational Transformation) or CRDTs, writing operations to an append-only log table first.
* **Deep Explanation**: This allows clients to fetch changes incrementally and merge concurrent writes using cell-level locking or Yjs algorithms.
* **Alternative Approach**: Storing the spreadsheet as a single large JSON blob.
  * *Pros*: Simple design, fast reads of the entire sheet.
  * *Cons*: Concurrent edits will overwrite each other, losing data.
* **Production Example**: Google Sheets database architectures.
* **Cross Questions**: *How do you index the Cell table?* We create a composite primary key on `(sheetId, row, col)`.

#### 5. How do you optimize query performance when executing deep relational joins (e.g., fetching workspaces, members, projects, boards, columns, and tasks in a single API call)?
* **Detailed Answer**: 
  1. We analyze the query plan using `EXPLAIN ANALYZE` to check for nested loops and table scans.
  2. Create composite indexes on foreign keys (e.g. `Project(workspaceId, id)`, `Column(boardId, id)`).
  3. Limit the depth of returned relations using select projections, avoiding fetching unused text fields (like descriptions or avatars).
  4. Implement application-level caching for static metadata.
* **Deep Explanation**: Deep JOIN queries can overwhelm the query planner, resulting in temporary disk writes. Fetching relations in two or three parallel, targeted queries can be faster than a single 6-way join statement.
* **Alternative Approach**: Denormalizing data into a single read-optimized NoSQL collection.
  * *Pros*: Fast $O(1)$ reads of the entire board state.
  * *Cons*: Complex synchronization logic required to sync updates across tables.
* **Production Example**: Dashboard query optimizations in project management tools.
* **Cross Questions**: *What is the difference between index-only scan and index scan?* An index-only scan fetches data directly from the index tree without loading table heap pages, which is faster.

#### 6. What is PostgreSQL index bloat, and how does it happen?
* **Detailed Answer**: Index bloat occurs when updates and deletes create dead row versions. In PostgreSQL, indexes contain pointers to these dead rows. Even after vacuuming, index pages on disk are not automatically shrunk, resulting in large indexes that consume unnecessary memory and slow down query lookups.
* **Why Interviewer Asks**: Explains why database performance degrades over time even with autovacuum enabled.
* **Common Mistakes**: Assuming autovacuum completely shrinks index sizes on disk.
* **Follow-up**: *How do you resolve index bloat without blocking writes?* By running `REINDEX INDEX CONCURRENTLY` in the background.
* **Production Example**: Reindexing databases periodically in cron jobs.

#### 7. How would you design a database replication failover mechanism that guarantees zero data loss (RPO = 0)?
* **Detailed Answer**: We implement **Synchronous Replication** to at least one standby replica. When a write transaction is executed, the primary database commits the change only after receiving confirmation that the transaction log (WAL) has been written to disk on the standby replica.
* **Deep Explanation**: 
  ```
  Client ---> Primary (Write) ---> Sync Replica (Write WAL) ---> Ack ---> Commit
  ```
  If the primary crashes before committing, the replica has the complete transaction log, allowing safe failover without data loss.
* **Alternative Approach**: Asynchronous replication.
  * *Pros*: High write throughput, as the primary commits immediately without waiting for replicas.
  * *Cons*: Risk of data loss if the primary crashes before WAL segments are synced.
* **Production Example**: High-availability setups for financial transaction systems.
* **Cross Questions**: *What is the drawback of synchronous replication?* Increased write latency, as transactions wait for network roundtrips to replicas.

#### 8. How would you implement a full-text search engine in PostgreSQL for tasks, comments, and project files?
* **Detailed Answer**: We use PostgreSQL's built-in full-text search features:
  1. Create a `tsvector` column on the search tables, combining search fields (e.g. title and description).
  2. Create a **GIN (Generalized Inverted Index)** index on the `tsvector` column.
  3. Query data using `to_tsquery` or `phraseto_tsquery` to match search strings.
* **Deep Explanation**: A GIN index maps words to the rows they appear in, allowing fast text search lookups. Using database triggers keeps the `tsvector` column updated on row changes.
* **Alternative Approach**: Offloading search to Elasticsearch or Algolia.
  * *Pros*: Rich features (typo tolerance, synonyms, facets) and fast execution.
  * *Cons*: Requires managing a separate search cluster and synchronization pipelines.
* **Production Example**: Search features in Git platforms or project boards.
* **Cross Questions**: *What is stemming in full-text search?* Reducing words to their base form (e.g. "running" to "run") to match related search terms.

#### 9. How do you handle schema changes that require data transformation in production?
* **Detailed Answer**: We write custom migration scripts running in a transaction:
  1. Add the new column as nullable.
  2. Run batch queries to populate the new column based on values in the old column.
  3. Deploy the application code that reads from the new column.
  4. Drop the old column once the system is stable.
* **Deep Explanation**: Avoid running lock-heavy updates on millions of rows in a single query. We update rows in batches (e.g., 5,000 rows per transaction) to keep database locks brief and prevent API timeouts.
* **Alternative Approach**: Offline migration during maintenance windows.
  * *Pros*: Simple, zero risk of intermediate data discrepancies.
  * *Cons*: Requires taking the platform offline, violating SLAs.
* **Production Example**: Standard database engineering pattern for zero-downtime upgrades.
* **Cross Questions**: *How do you rollback if a batch fails?* The batch transaction rolls back, and we log the last processed ID to resume.

#### 10. How would you design a database schema to support workspace billing subscriptions, metered usage, and invoice credits?
* **Detailed Answer**: We split the schema into four tables:
  1. `Subscription`: Tracks workspace subscriptions (tier, status, billing cycle, price).
  2. `MeteredUsage`: Records resource consumption events (e.g. active users, storage used).
  3. `CreditBalance`: Tracks ledger credits (amount, currency, source, expiry).
  4. `Invoice`: Records billing transactions, listing line items, tax, and payment status.
* **Deep Explanation**: We use a ledger-based design for credits, writing modifications as append-only entries (debit/credit) instead of mutating a single balance column. This preserves audit trails and prevents balance drift.
* **Alternative Approach**: Storing active balances directly on the Workspace table.
  * *Pros*: Simple queries to check balance limits.
  * *Cons*: Lacks audit history, making it difficult to debug billing discrepancies.
* **Production Example**: Stripe-like billing system integrations.
* **Cross Questions**: *How do you prevent double-spending of credits?* By checking and writing ledger balances within a Serializable transaction.

#### 11. What is index selectivity, and how does it affect query planning?
* **Detailed Answer**: Selectivity is the ratio of unique values in a column to the total number of rows. High selectivity columns (like email or UUIDs) make indexes highly effective. Low selectivity columns (like boolean flags or status enums) make indexes ineffective, as the database planner will fallback to table scans.
* **Why Interviewer Asks**: Tests indexing optimization theory.
* **Common Mistakes**: Creating indexes on boolean fields (e.g., `isDone`).
* **Follow-up**: *What is a partial index?* An index created on a subset of rows using a WHERE clause, which reduces index size and improves selectivity.
* **Production Example**: Creating a partial index on `Task(dueDate) WHERE dueDate IS NOT NULL`.

#### 12. How do you handle database failover in a Kubernetes deployment?
* **Detailed Answer**: We use database operators (like CloudNativePG or Zalando Postgres Operator) that deploy PostgreSQL in a StateFulSet. The operator monitors nodes; if the primary database fails, it automatically promotes a standby replica to primary and updates Kubernetes service endpoints to route traffic correctly.
* **Why Interviewer Asks**: Evaluates high-availability cloud architecture design.
* **Common Mistakes**: Storing database files inside standard ephemeral container storage, leading to data loss on pod restarts.
* **Follow-up**: *What is a split-brain scenario?* A state where two database replicas both believe they are the primary, leading to database discrepancies.
* **Production Example**: Running highly available PostgreSQL clusters in production EKS/GKE environments.

#### 13. How do you design a database backup strategy for compliance audits?
* **Detailed Answer**: 
  1. Configure automated daily logical backups (`pg_dump`) and physical backups (WAL segments).
  2. Store backups in encrypted, write-once-read-many (WORM) storage (like AWS S3 Object Lock) to prevent deletion.
  3. Set retention policies to match compliance requirements (e.g. 7 years).
  4. Run automated recovery drills monthly to verify backup integrity.
* **Why Interviewer Asks**: Backups are useless if they cannot be restored. Testing recovery is critical.
* **Common Mistakes**: Storing database backups on the same physical server hosting the database.
* **Follow-up**: *What is the difference between logical and physical backups?* Logical backups store SQL statements to rebuild schemas and data. Physical backups copy raw database files, which is faster to restore.
* **Production Example**: Compliance backup pipelines in financial or healthcare platforms.

#### 14. What are PostgreSQL CTEs (Common Table Expressions), and how do they perform?
* **Detailed Answer**: CTEs are temporary result sets defined within a query using the `WITH` clause. In older PostgreSQL versions (before v12), CTEs acted as optimization fences, meaning the database materialized the CTE results, bypassing planner optimizations. In modern versions, CTEs are inlined by default unless marked `MATERIALIZED`.
* **Why Interviewer Asks**: Explains query optimization history and performance differences.
* **Common Mistakes**: Assuming CTEs are always optimized identically to subqueries.
* **Follow-up**: *What is a recursive CTE?* A CTE that references itself, used to query hierarchical data structures (like folder trees or task dependencies).
* **Production Example**: Querying nested task checklist trees.

#### 15. How do you design a database schema to support custom task fields?
* **Detailed Answer**: We use the **EAV (Entity-Attribute-Value)** pattern or a **JSONB** column.
  1. *JSONB*: Store custom fields as key-value pairs in a single `customFields` JSONB column on the `Task` table.
     * *Pros*: Simple queries, fast reads, and support for nested data.
     * *Cons*: Limited validation constraints and slower index-based searches.
  2. *EAV*: Create `TaskField`, `TaskFieldDefinition`, and `TaskFieldValue` tables.
     * *Pros*: Strict type validation and index-based searches.
     * *Cons*: Requires complex queries with multiple joins.
* **Deep Explanation**: We choose JSONB for custom fields, utilizing PostgreSQL GIN indexes on JSONB keys to optimize search queries.
* **Alternative Approach**: Dynamically running DDL statements (`ALTER TABLE ADD COLUMN`) for each custom field.
  * *Pros*: Normal database structure.
  * *Cons*: Dangerous in production, as running DDL statements concurrently locks tables and crashes database engines.
* **Production Example**: Custom fields configurations in Jira or Asana.
* **Cross Questions**: *What is the difference between JSON and JSONB in PostgreSQL?* JSON stores text representation directly, requiring parsing on read. JSONB stores data in a binary format, which is slightly slower to write but significantly faster to query and index.

#### 16. How would you monitor database query performance in production?
* **Detailed Answer**: We enable PostgreSQL's `pg_stat_statements` extension to track execution statistics for all queries. We monitor metrics like total execution time, call frequency, and average read/write times, routing alerts to monitoring dashboards (like Datadog or Prometheus/Grafana).
* **Why Interviewer Asks**: Observability is critical for detecting performance bottlenecks before they cause downtime.
* **Common Mistakes**: Relying only on application logs to track slow queries.
* **Follow-up**: *What is query plan cache?* A cache of execution plans for prepared statements, reducing query parse times.
* **Production Example**: Setting up alerts for queries executing in over 500ms.

#### 17. How do you design a database schema to support system notifications with multiple delivery channels?
* **Detailed Answer**: We create a `Notification` table that stores the message payload, user references, and delivery states.
  ```
  Notification(id, userId, title, body, status)
  NotificationChannel(id, notificationId, channelType [Email, Push, SMS], status [Pending, Sent, Failed], sentAt)
  ```
* **Why Interviewer Asks**: Separating notification creation from delivery channels prevents delivery failures from blocking notification logs.
* **Common Mistakes**: Storing delivery details as flags directly on the Notification table, which fails to support retries or multiple channels.
* **Follow-up**: *How do you scale notification delivery?* By pushing notifications to a message queue for processing by background workers.
* **Production Example**: Notification engines in modern SaaS platforms.

#### 18. How do you design a database schema to support user invite links with varying scopes and validation limits?
* **Detailed Answer**: 
  We create a `WorkspaceInvite` table:
  * `id`: UUID (Primary Key)
  * `workspaceId`: UUID (References Workspace)
  * `email`: String (Optional - Null indicates a shareable link)
  * `token`: String (Unique index)
  * `role`: Enum (Member role to assign)
  * `expiresAt`: DateTime (Optional expiry date)
  * `maxUses`: Integer (Optional - Max times the link can be used)
  * `useCount`: Integer (Counter of times the link has been used)
* **Why Interviewer Asks**: Tests validation design and concurrency checks (e.g. preventing race conditions where users exceed `maxUses`).
* **Common Mistakes**: Storing invite tokens as plain text in logs or neglecting check constraints on usage limits.
* **Follow-up**: *How do you increment useCount safely?* Within a transaction using `UPDATE WorkspaceInvite SET useCount = useCount + 1 WHERE id = X AND useCount < maxUses`.

#### 19. How do you design a database schema to support task dependency tracking (e.g. Task A blocks Task B)?
* **Detailed Answer**: We create a self-referencing relationship using a `TaskDependency` table:
  * `id`: UUID
  * `blockedTaskId`: UUID (References Task)
  * `blockingTaskId`: UUID (References Task)
  * `type`: Enum (e.g., `BLOCKS`, `RELATED_TO`)
* **Why Interviewer Asks**: Hierarchical modeling is common in project boards.
* **Common Mistakes**: Storing dependencies as arrays inside the Task table, which makes it difficult to detect circular dependencies.
* **Follow-up**: *How do you prevent circular dependencies (e.g. A blocks B, and B blocks A)?* Before inserting a dependency, run a recursive query to ensure no path exists from the blocked task back to the blocking task.
* **Production Example**: Dependency tracking features in Jira.

#### 20. How would you handle hot-standby read replica sync delays when generating billing invoices?
* **Detailed Answer**: Read replicas are asynchronously synced, meaning query results can be slightly stale (replica lag). For critical operations like compiling billing invoices, we bypass read replicas and route read queries directly to the primary database to ensure we read the latest time logs.
* **Why Interviewer Asks**: Using stale data for financial transactions leads to billing discrepancies and customer disputes.
* **Common Mistakes**: Routing all read queries to replicas without evaluating data consistency requirements.
* **Follow-up**: *What header can you set to bypass replica caches?* We can write query wrappers that direct traffic to different pool instances based on request flags.
* **Production Example**: Routing billing compiles or critical setting updates to primary database connections.
